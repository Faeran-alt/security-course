В ходе работы была создана личная криптографическая ключевая пара для использования в системе GnuPG.

* Key-ID: 3F8C7653ADC51EAF
* Тип ключа: RSA 4096 бит
* Email: dianazakh2006@gmail.com
* Размер публичного ключа: зависит от результата экспорта ключа

### Что означает флаг --armor

Флаг `--armor` переводит бинарные данные GPG в текстовый ASCII-формат. В результате ключ, подпись или зашифрованное сообщение сохраняются в виде обычного текста с заголовками:

-----BEGIN PGP PUBLIC KEY BLOCK-----

или

-----BEGIN PGP MESSAGE-----

Такой формат удобен для передачи через электронную почту, мессенджеры и публикации на GitHub.

## 2. Шифрование

Для проверки работы асимметричного шифрования был создан файл с конфиденциальными данными и зашифрован собственным публичным ключом.

### Зашифрованный файл (фрагмент)

```text
-----BEGIN PGP MESSAGE-----
...
```

### Можно ли прочитать исходные данные из зашифрованного файла

Нет. После шифрования содержимое преобразуется в криптографически защищённый набор символов. Без соответствующего приватного ключа восстановить исходную информацию невозможно.

### Что произойдёт при вводе неправильного пароля

Если ввести неверный пароль от приватного ключа, GPG не сможет получить доступ к ключу расшифрования. Процесс завершится ошибкой и содержимое файла останется недоступным.

### Почему нельзя расшифровать файл, зашифрованный для другого пользователя

При шифровании используется публичный ключ конкретного получателя. Расшифровка возможна только при наличии соответствующего приватного ключа. Даже автор файла не сможет прочитать данные, если они были зашифрованы исключительно для другого пользователя.

## 3. Цифровая подпись

Для проверки механизма электронной подписи был создан файл и отдельная подпись с использованием GPG.

### Вывод проверки подписи

```text
gpg: Good signature from "milimika <dianazakh2006@gmail.com>"
```

Сообщение Good signature означает, что файл не изменялся после подписания и действительно был подписан владельцем указанного ключа.

### После изменения файла

```text
gpg: BAD signature from "milimika <dianazakh2006@gmail.com>"
```

После изменения содержимого документа проверка завершилась сообщением BAD signature.

### Почему подпись стала недействительной

При создании подписи GPG вычисляет криптографический хэш содержимого файла. Любое изменение документа приводит к изменению хэша. В результате подпись больше не соответствует данным файла и считается недействительной.

### Что проверяет команда gpg --verify

Команда выполняет проверку:

* подлинности автора документа;
* целостности данных;
* отсутствия изменений после подписания;
* соответствия подписи открытому ключу владельца.

## 5. Сравнение rm и shred

| Операция | Что происходит с данными на диске                                | Можно восстановить?            |
| -------- | ---------------------------------------------------------------- | ------------------------------ |
| rm       | Удаляется запись о файле в файловой системе                      | Да, пока блоки не перезаписаны |
| shred    | Содержимое файла многократно перезаписывается случайными данными | Практически невозможно         |

### Почему данные могут сохраниться после rm

Команда `rm` не уничтожает содержимое файла физически. Она удаляет только информацию о расположении файла в файловой системе. Пока свободные блоки не будут использованы повторно, данные могут быть восстановлены специализированными программами.

### Что делает shred

Команда `shred` несколько раз перезаписывает содержимое файла случайными данными и нулями, после чего удаляет его. Это значительно усложняет или делает невозможным восстановление информации.

### Когда необходимо затирать свободное место

Подобная процедура может потребоваться:

* перед продажей компьютера;
* перед утилизацией накопителя;
* после удаления конфиденциальных документов;
* в организациях, работающих с персональными данными и коммерческой тайной.

## 7. Контрольные вопросы

## 1. В чём разница между шифрованием и подписью? Можно ли одновременно зашифровать и подписать файл?

Шифрование обеспечивает конфиденциальность информации и делает данные недоступными для посторонних лиц. Цифровая подпись подтверждает авторство и целостность данных, но не скрывает их содержимое. Один и тот же файл можно одновременно подписать и зашифровать, обеспечив как защиту содержимого, так и подтверждение подлинности автора.

---

## 2. Почему публичный ключ можно раздавать всем, а приватный нельзя никому?

Публичный ключ предназначен для шифрования данных и проверки подписей, поэтому его распространение не создаёт угрозы безопасности. Приватный ключ используется для расшифровки сообщений и создания цифровых подписей. Получив приватный ключ, злоумышленник сможет выдавать себя за владельца и получать доступ к защищённым данным.

---

## 3. Что произойдёт с зашифрованными данными, если потерять приватный ключ?

Без приватного ключа расшифровать данные будет невозможно. Если резервная копия ключа отсутствует, доступ к информации будет потерян навсегда независимо от используемого оборудования или программного обеспечения.

---

## 4. Можно ли подделать Verified-коммит на GitHub без приватного GPG-ключа владельца?

Нет. Для создания корректной цифровой подписи необходим соответствующий приватный ключ. Без него невозможно сформировать подпись, которая будет успешно проверена GitHub и получит статус Verified.

---

## 5. Почему shred плохо работает на SSD? Что использовать вместо него?

SSD используют механизм выравнивания износа (Wear Leveling), поэтому данные могут физически записываться в другие области памяти, а старые копии временно сохраняться. Из-за этого shred не гарантирует полное уничтожение информации. Для SSD рекомендуется использовать встроенные функции Secure Erase или полное шифрование накопителя с последующим уничтожением ключей шифрования.

---

## 6. Коллега говорит: «Я отформатировал диск перед продажей, значит всё безопасно». Он прав?

Нет. Быстрое форматирование обычно удаляет только служебную информацию файловой системы, но не сами данные. Многие файлы после такого форматирования могут быть восстановлены специализированными программами. Для надёжного удаления необходимо использовать Secure Erase, полную перезапись накопителя или предварительное шифрование диска.

---

## 7. Чем отличается detach-sign (--detach-sign) от встроенной подписи (--sign)?

При использовании --detach-sign подпись сохраняется в отдельном файле и исходный документ остаётся неизменным. При использовании --sign подпись встраивается непосредственно в результирующий файл. Отдельная подпись удобна для проверки оригинальных файлов без их изменения.
