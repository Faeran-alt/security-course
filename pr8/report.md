## 2. Обнаружение подозрительных процессов

Для поиска подозрительных процессов использовалась команда:

ps aux | grep '/tmp'

Были обнаружены процессы, запущенные из каталога /tmp/.hidden_malware. Подобное расположение файлов считается подозрительным, поскольку временные каталоги часто используются вредоносным программным обеспечением для сокрытия своей деятельности.

Дополнительно было исследовано дерево процессов с помощью команды:

ps auxf

Это позволило установить взаимосвязь между процессами backdoor.sh, listener.sh и их дочерними процессами.

## 3. Анализ сетевых соединений

Для поиска открытых портов использовалась команда:

ss -tulnp

В результате был обнаружен открытый TCP-порт 4444, прослушиваемый процессом nc.

Дополнительно была выполнена команда:

sudo lsof -i :4444

Утилита lsof позволила установить связь между сетевым портом, процессом и исполняемым файлом. Таким образом был подтверждён запуск имитации backdoor-службы.

### Cron

В пользовательском cron были обнаружены следующие записи:

Записи обеспечивали автоматический запуск скрипта после загрузки системы и каждые пять минут.

### Systemd

Был обнаружен пользовательский сервис:

Содержимое unit-файла:

Сервис автоматически перезапускал скрипт при его завершении, обеспечивая устойчивость работы.

### Bashrc

Данная команда запускала скрипт при каждом открытии терминала пользователем.

## 10. Контрольные вопросы

## 1. Назовите пять мест в Linux где вредонос может прописать автозапуск. Чем они отличаются по уровню привилегий?

1. Cron (crontab) — позволяет запускать задачи по расписанию или при загрузке системы. Пользовательские задания работают от имени пользователя, системные могут работать от имени root.

2. Systemd-сервисы (/etc/systemd/system/) — системные сервисы запускаются с повышенными привилегиями и требуют права администратора для создания и изменения.

3. Пользовательские systemd-сервисы (~/.config/systemd/user/) — создаются обычным пользователем и работают без прав root, но обеспечивают автозапуск после входа пользователя в систему.

4. Файлы профиля оболочки (~/.bashrc, ~/.profile, ~/.bash_profile) — выполняются при входе пользователя в систему или запуске терминала и работают от имени пользователя.

5. Файл /etc/rc.local — выполняется во время загрузки системы и обычно работает с правами root, поэтому позволяет запускать процессы с максимальными привилегиями.

Основное различие заключается в том, что часть механизмов работает от имени обычного пользователя, а часть требует административных прав и может запускать процессы с привилегиями root.

## 2. Чем lsof отличается от ss? Когда использовать каждый?

lsof (List Open Files) показывает открытые файлы, сокеты, сетевые соединения и другие ресурсы, используемые процессами. С его помощью можно определить, какие файлы или порты использует конкретный процесс.

ss (Socket Statistics) предназначен для анализа сетевых соединений и открытых портов. Он показывает, какие процессы слушают порты и какие сетевые подключения активны.

Использовать ss удобно для поиска подозрительных сетевых соединений и открытых портов. lsof применяется для более глубокого анализа процесса и его открытых файлов, сокетов и ресурсов.

## 3. Что означает строка (deleted) в выводе lsof? Почему вредоносы используют этот трюк?

Строка (deleted) означает, что файл был удалён из файловой системы, однако процесс продолжает использовать его через уже открытый файловый дескриптор.

Вредоносное программное обеспечение использует этот приём для сокрытия своих файлов. После запуска исполняемый файл удаляется, поэтому его невозможно увидеть обычными средствами просмотра файловой системы. Однако процесс продолжает работать до завершения и остаётся видимым через lsof.

## 4. Пользовательский systemd-сервис в ~/.config/systemd/user/ — нужен ли root чтобы его создать? Почему это опасно?

Для создания пользовательского systemd-сервиса права root не требуются. Любой пользователь может создать unit-файл в каталоге ~/.config/systemd/user/ и включить его через systemctl --user enable.

Это опасно потому, что злоумышленник, получивший доступ к учётной записи пользователя, может обеспечить постоянный запуск вредоносного ПО после каждой авторизации без получения привилегий администратора.

## 5. Вредонос переименовал себя в systemd-journald. Как отличить настоящий системный процесс от поддельного?

Необходимо проверить путь запуска процесса и его параметры.

Для этого можно использовать команды:

ps aux | grep systemd-journald
sudo lsof -p PID
readlink /proc/PID/exe

Настоящий systemd-journald запускается из системных каталогов, например:

/usr/lib/systemd/systemd-journald

Если процесс с таким именем запущен из /tmp, /var/tmp, домашней директории пользователя или скрытого каталога, это является признаком маскировки вредоносного ПО.
