# ПР №7. AppArmor, Capabilities и Docker

## 1. Linux Capabilities

## Разбор getcap /usr/bin/ping

Команда:

```bash
getcap /usr/bin/ping
```

Результат:

```bash
cap_net_raw=ep
```

Расшифровка:

- `cap_net_raw` — разрешает создание raw-сокетов и работу с низкоуровневыми сетевыми пакетами.
- `e` (effective) — capability активна во время выполнения процесса.
- `p` (permitted) — capability входит в разрешённый набор процесса.

---

## Capabilities в системе

Для поиска capabilities использовались команды:

```bash
getcap /usr/bin/python3* 2>/dev/null
sudo find /usr -xdev 2>/dev/null | xargs getcap 2>/dev/null
```

Были обнаружены файлы с назначенными capabilities, включая:

- ping;
- mount;
- umount;
- системные утилиты, требующие ограниченных привилегий.

Использование capabilities позволяет выдавать программе только необходимые права без запуска её от имени root.

---

## Capabilities текущего процесса

Команда:

```bash
cat /proc/self/status | grep -i cap
```

Основные поля:

- `CapPrm` — разрешённый набор capabilities процесса.
- `CapEff` — активный набор capabilities.
- `CapBnd` — максимальный набор capabilities, который процесс может получить.

Для расшифровки использовалась команда:

```bash
capsh --decode=ЗНАЧЕНИЕ
```

У обычного shell отсутствуют привилегированные capabilities, что соответствует принципу наименьших привилегий.

---

## Демонстрация setcap

### До назначения capability

```bash
python3 /tmp/test-port.py 80
```

Привязка к порту 80 была запрещена.

```bash
python3 /tmp/test-port.py 8080
```

Привязка к порту 8080 выполнялась успешно.

### После назначения capability

Команды:

```bash
sudo setcap cap_net_bind_service=ep $PYBIN
getcap $PYBIN
```

После назначения capability интерпретатор Python смог открывать порт 80 без использования sudo.

Это безопаснее запуска через sudo, потому что:

- процесс не получает полный root-доступ;
- выдаётся только одна необходимая capability;
- снижается риск компрометации системы.

После завершения проверки capability была удалена:

```bash
sudo setcap -r $PYBIN
```

---

## Флаги e, i, p

Команда:

```bash
sudo capsh --caps='cap_net_raw+eip'
```

Расшифровка:

- `e` — effective;
- `i` — inheritable;
- `p` — permitted.

`ep` предоставляет capability только текущему процессу.

`eip` дополнительно позволяет наследовать capability дочерними процессами.

## 2. AppArmor

## Количество профилей

Команда:

```bash
sudo aa-status
```

В системе присутствовали профили в режимах:

- enforce;
- complain.

AppArmor реализует механизм MAC (Mandatory Access Control), ограничивая действия программ независимо от прав пользователя.

---

## Анализ существующих профилей

Для анализа использовались команды:

```bash
ls /etc/apparmor.d/
sudo cat /etc/apparmor.d/usr.bin.man
```

Профиль программы man содержит разрешения на чтение документации, запуск вспомогательных программ и доступ к необходимым системным файлам.

Основные обозначения:

- `r` — чтение;
- `w` — запись;
- `ix` — выполнение программы.

---

## Профиль pr07-reader

Был создан тестовый скрипт, выполняющий:

- чтение разрешённого файла;
- чтение файла `/etc/shadow`;
- запись в каталог `/tmp`;
- запись в каталог `/etc`.

---

## Результаты работы профиля

| Действие | Без профиля | complain | enforce |
|-----------|-----------|-----------|-----------|
| Читать /tmp/pr07-allowed.txt | Разрешено | Разрешено | Разрешено |
| Читать /etc/shadow | Разрешено | Разрешено + запись в журнал | Запрещено |
| Писать в /tmp/pr07-output.txt | Разрешено | Разрешено | Разрешено |
| Писать в /etc/ | Запрещено системой | Фиксируется в журнале | Запрещено |

В режиме complain AppArmor только регистрирует нарушения.

В режиме enforce нарушения блокируются согласно профилю безопасности.

---

## Разбор строки DENIED

Пример записи:

```text
apparmor="DENIED" operation="mknod" profile="/usr/local/bin/pr07-reader" name="/etc/pr07-hack.txt" pid=3037 comm="pr07-reader" requested_mask="c" denied_mask="c"
```

Расшифровка:

- `apparmor="DENIED"` — действие заблокировано.
- `operation="mknod"` — попытка создать файл.
- `profile=` — профиль AppArmor.
- `name=` — путь к объекту.
- `pid=` — идентификатор процесса.
- `comm=` — имя процесса.
- `requested_mask=` — запрошенная операция.
- `denied_mask=` — операция, запрещённая профилем.

## 3. Docker — изоляция

## Сравнение хоста и контейнера

| Ресурс | Хост | Контейнер |
|----------|----------|----------|
| Процессы | Все процессы системы | Только процессы контейнера |
| Сетевые интерфейсы | ens33, lo, docker0 | lo, eth0 |
| Корневая файловая система | Полная система | Изолированная файловая система |
| /etc/shadow хоста | Доступен | Недоступен |
| Монтирование ФС | Разрешено root | Ограничено |

Контейнеры используют namespaces и cgroups, благодаря чему процессы видят только собственное окружение.

---

## Volumes

Был создан volume:

```bash
-v /tmp/pr07-data:/data
```

Контейнер получил доступ только к каталогу `/data`.

Файл:

```text
/tmp/pr07-secret.txt
```

оставался недоступным.

Такой способ безопаснее использования `--privileged`, так как предоставляет доступ только к необходимым данным.

---

## Capabilities контейнера

### Обычный контейнер

```text
CapEff: 00000000a80425fb
```

Контейнер имеет ограниченный набор возможностей.

### Контейнер с --privileged

```text
CapEff: 0000003fffffffff
```

Контейнер получает практически все capabilities Linux.

Отсутствующие в обычном контейнере возможности:

- SYS_ADMIN;
- SYS_MODULE;
- SYS_PTRACE;
- NET_ADMIN;
- SYS_TIME;
- SYS_BOOT и другие.

Параметр `--privileged` опасен, поскольку контейнер получает права, близкие к root-пользователю хоста.

---

## Запуск без NET_RAW

Команда:

```bash
docker run --rm --cap-drop ALL ubuntu:22.04 ping -c 1 ya.ru
```

Ping завершался ошибкой.

После добавления capability:

```bash
docker run --rm --cap-drop ALL --cap-add NET_RAW alpine ping -c 1 ya.ru
```

утилита работала успешно.

Это демонстрирует принцип наименьших привилегий.

---

## Запуск не от root

По умолчанию контейнер запускался от root:

```bash
docker run --rm ubuntu:22.04 whoami
```

Результат:

```text
root
```

После запуска:

```bash
docker run --rm --user 1000:1000 ubuntu:22.04 id
```

контейнер работал от непривилегированного пользователя.

Попытка установки пакетов завершалась ошибкой недостатка прав.

Использование непривилегированного пользователя снижает последствия возможного взлома контейнера.

## 4. Итоговое задание — nginx с ограничениями

## Запуск nginx

Контейнер запускался командой:

```bash
docker run -d \
--name pr07-nginx \
--cap-drop ALL \
--cap-add NET_BIND_SERVICE \
--cap-add CHOWN \
--cap-add DAC_OVERRIDE \
--cap-add SETGID \
--cap-add SETUID \
-p 8080:80 \
nginx:alpine
```

Проверка работы:

```bash
curl http://localhost:8080 | head -5
```

Результат:

```html
<!DOCTYPE html>
<html>
<head>
<title>Welcome to nginx!</title>
```

---

## Capabilities nginx

Команда:

```bash
docker exec pr07-nginx cat /proc/1/status | grep Cap
```

Результат:

```text

```

Используемые capabilities:

- NET_BIND_SERVICE;
- CHOWN;
- DAC_OVERRIDE;
- SETGID;
- SETUID.

Они необходимы для:

- открытия порта 80;
- смены пользователя процесса;
- работы с файлами и правами доступа.

## 5. Эшелонированная защита

| Слой | Инструмент | Что ограничивает |
|--------|--------|--------|
| DAC | chmod/chown | Права пользователей на файлы |
| Capabilities | cap-drop / cap-add | Привилегии процессов |
| MAC | AppArmor | Доступ программ к ресурсам |
| Изоляция | Docker namespaces и cgroups | Процессы, сеть, файловую систему и ресурсы |

## 6. Выводы

В ходе практической работы были изучены механизмы ограничения привилегий и изоляции процессов в Linux.

Linux Capabilities позволяют выдавать процессам только необходимые права без использования полного root-доступа. AppArmor реализует мандатный контроль доступа и ограничивает действия приложений даже в случае их компрометации. Docker обеспечивает изоляцию процессов, сети, файловой системы и ресурсов за счёт namespaces и cgroups.

Совместное использование DAC, Capabilities, AppArmor и контейнеризации реализует принцип эшелонированной защиты и существенно снижает последствия возможных атак на систему.

## 7. Контрольные вопросы

## 1. Чем DAC отличается от MAC? Почему одного DAC недостаточно?

DAC (Discretionary Access Control) позволяет владельцу объекта самостоятельно назначать права доступа с помощью chmod и chown.

MAC (Mandatory Access Control) задаёт правила доступа централизованно и не зависит от владельца объекта.

Одного DAC недостаточно, поскольку после компрометации учётной записи злоумышленник получает все права пользователя. MAC дополнительно ограничивает действия даже легально запущенных процессов.

---

## 2. Что означает запись cap_net_bind_service=eip? Чем ep отличается от eip?

cap_net_bind_service позволяет открывать привилегированные порты ниже 1024 без использования root.

- e (effective) — capability активна;
- i (inheritable) — capability наследуется дочерними процессами;
- p (permitted) — capability разрешена процессу.

ep действует только для текущего процесса, а eip позволяет дополнительно наследовать capability дочерним процессам.

---

## 3. В чём разница между complain и enforce в AppArmor? Зачем нужен complain?

Режим complain только регистрирует нарушения политики безопасности и записывает их в журнал.

Режим enforce реально блокирует запрещённые действия.

Complain используется для тестирования и настройки профилей AppArmor перед переводом в рабочий режим.

---

## 4. Docker использует то же ядро что и хост. Почему тогда контейнер изолирован?

Изоляция достигается с помощью:

- namespaces;
- cgroups;
- capabilities;
- AppArmor.

Благодаря этому контейнер видит только собственные процессы, сеть и файловую систему.

---

## 5. Злоумышленник нашёл RCE в nginx. Что он может и не может сделать?

Может:

- выполнять команды от имени процесса nginx;
- читать доступные приложению файлы;
- взаимодействовать с сетевыми сервисами.

Не может:

- получить полный root-доступ к хосту;
- монтировать файловые системы;
- управлять устройствами;
- обходить ограничения AppArmor;
- использовать отсутствующие capabilities.

Используемые механизмы значительно уменьшают последствия успешной атаки.

---

## 6. Почему --privileged является антипаттерном? Когда он может быть оправдан?

Параметр --privileged выдаёт контейнеру практически все capabilities Linux и снимает большинство ограничений безопасности.

В случае компрометации контейнера злоумышленник получает возможности, близкие к root-пользователю хоста.

Использование может быть оправдано только для:

- разработки системного ПО;
- тестирования драйверов;
- работы с оборудованием;
- специализированных лабораторных стендов.

Для обычных сервисов рекомендуется использовать принцип наименьших привилегий и выдавать только необходимые права.
