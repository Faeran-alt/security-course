# ПР №9. Лицензирование и open-source compliance

---

# 1. Выбор лицензии

Выбранная лицензия: MIT

Тип лицензии: разрешительная (permissive license).

Лицензия MIT является одной из наиболее популярных открытых лицензий благодаря своей простоте и минимальному количеству ограничений.

Что разрешает лицензия MIT:

* использовать программное обеспечение в любых целях;
* изменять исходный код;
* создавать производные продукты;
* распространять оригинальные и изменённые версии;
* использовать программное обеспечение в коммерческих проектах;
* включать код в закрытые проприетарные продукты.

Что требует лицензия MIT:

* сохранять уведомление об авторских правах;
* сохранять текст лицензии при распространении программного обеспечения.

Может ли кто-то продавать мой код:

Да. Лицензия MIT разрешает коммерческое использование программного обеспечения, включая его продажу, при условии сохранения уведомления об авторских правах и текста лицензии.

# 2. Результат licensee detect

Вывод команды:

```text
License:        MIT
Matched files:  LICENSE

LICENSE:
  Attribution:   Copyright (c) 2024 milimika77
  Confidence:    100.00%
  Matcher:       Licensee::Matchers::Exact
  License:       MIT
```

Что означает Match % (Confidence):

Показатель Confidence показывает степень совпадения содержимого файла LICENSE с эталонным шаблоном лицензии из базы Licensee. Значение 100% означает полное совпадение с лицензией MIT.

Откуда GitHub знает какая лицензия используется в репозитории:

GitHub использует инструмент Licensee, который автоматически анализирует файл LICENSE в корне репозитория и сравнивает его содержимое с базой известных лицензий. Если совпадение найдено, GitHub отображает лицензию на странице репозитория.

# 3. SPDX-заголовки

Пример добавленного заголовка:

```python
# SPDX-FileCopyrightText: 2024 Мили Мика
# SPDX-License-Identifier: MIT
```

Назначение строк:

SPDX-FileCopyrightText содержит сведения об авторе и владельце авторских прав.

SPDX-License-Identifier содержит стандартизированное машиночитаемое обозначение лицензии.

Использование SPDX позволяет автоматически анализировать лицензии файлов различными инструментами проверки compliance.

Результат reuse lint:

После добавления SPDX-заголовков инструмент REUSE корректно определил лицензионную информацию в размеченных файлах. Однако проект не был полностью совместим со спецификацией REUSE, поскольку большинство файлов предыдущих практических работ не содержали соответствующих SPDX-заголовков.

Сообщение:

```text
Unfortunately, your project is not compliant with version 3.3 of the REUSE Specification :-(
```

означает, что в проекте ещё присутствуют файлы без лицензионной информации.

# 6. Анализ совместимости лицензий

Сценарий 1. Проект под MIT использует библиотеку под GPL-3.0.

Включить такую библиотеку в закрытый коммерческий продукт нельзя. GPL требует распространения производного программного обеспечения на условиях GPL с раскрытием исходного кода.

Сценарий 2. Проект под Apache-2.0 использует библиотеку под MIT.

Да, использование возможно. Лицензия MIT совместима с Apache-2.0.

Сценарий 3. Компания использует MIT-код без указания авторства.

Да, это нарушение лицензии MIT, поскольку лицензия требует сохранения уведомления об авторских правах и текста лицензии.

Сценарий 4. Использование AGPL-библиотеки в SaaS без раскрытия исходников.

Нельзя. AGPL требует предоставления исходного кода пользователям, взаимодействующим с программой через сеть.

Сценарий 5. В репозитории отсутствует файл LICENSE.

Использовать такой код нельзя. При отсутствии лицензии авторские права сохраняются автоматически, а разрешение на использование, изменение и распространение отсутствует.

# Контрольные вопросы

## 1. Чем разрешительная лицензия (MIT) отличается от копилефтной (GPL)? Что значит «вирусный эффект» GPL?

MIT является разрешительной лицензией и позволяет использовать код практически без ограничений, включая коммерческие и закрытые проекты. GPL относится к копилефтным лицензиям и требует, чтобы производные работы распространялись под той же лицензией. Вирусный эффект GPL означает, что производное программное обеспечение также должно распространяться под GPL.

## 2. Что такое SPDX-идентификатор? Зачем нужна машиночитаемая форма лицензии?

SPDX-идентификатор представляет собой стандартизированное краткое обозначение лицензии, например MIT или GPL-3.0-only. Машиночитаемая форма позволяет автоматически определять лицензии и выполнять проверки compliance.

## 3. Компания встроила GPL-2.0 библиотеку в закрытый коммерческий продукт. Что нарушено и каковы последствия?

Нарушены требования лицензии GPL-2.0 о распространении производного программного обеспечения под GPL и раскрытии исходного кода. Возможны требования прекратить распространение продукта либо раскрыть исходный код.

## 4. Что такое SBOM и зачем он нужен?

SBOM (Software Bill of Materials) представляет собой перечень всех компонентов программного обеспечения. Он используется для контроля лицензий, анализа зависимостей и быстрого выявления уязвимых компонентов.

## 5. Репозиторий на GitHub без LICENSE-файла. Можно ли использовать код?

Нет. Отсутствие лицензии означает отсутствие разрешения на использование, изменение и распространение программного обеспечения.

## 6. Чем отличается AGPL от GPL? Какую лазейку она закрывает?

GPL требует раскрытия исходников при распространении программного обеспечения. AGPL дополнительно требует раскрытия исходного кода при предоставлении доступа к программе через сеть. Это устраняет возможность использовать программу как SaaS без публикации исходников.

## 7. Зачем проверять лицензии зависимостей в CI?

Автоматическая проверка позволяет своевременно обнаруживать появление несовместимых лицензий, предотвращать нарушения лицензионных требований и контролировать изменения зависимостей на протяжении всего жизненного цикла проекта.
