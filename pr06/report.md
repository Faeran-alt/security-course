# ПР №6. Анализ сетевого трафика: tcpdump и Wireshark

## 1. Разбор строки tcpdump

Строка:
`12:34:56.789012 IP 127.0.0.1.44321 > 127.0.0.1.8080: Flags [S], seq 123, length 0`

| Поле | Значение | Что означает |
|------|------|------|
| 12:34:56.789012 | Время | Время захвата пакета |
| IP | Протокол IP | Используется протокол IPv4 |
| 127.0.0.1.44321 | IP и порт отправителя | Локальный адрес и случайный порт клиента |
| 127.0.0.1.8080 | IP и порт назначения | Локальный HTTP-сервер на порту 8080 |
| Flags [S] | SYN-флаг | Начало TCP-соединения |
| seq 123 | Sequence number | Номер последовательности TCP |
| length 0 | Длина данных | В пакете нет полезной нагрузки |

## 2. Перехваченные данные HTTP

Во время анализа HTTP-трафика было обнаружено, что данные передаются в открытом виде без шифрования. В выводе tcpdump были видны логин, пароль и заголовок Authorization.

Пример перехваченных данных:

`username=vasya&password=MyPassword2024`

`Authorization: Basic dmFzeWE6TXlQYXNzd29yZDIwMjQ=`

Это опасно, потому что любой злоумышленник, имеющий доступ к сети, может перехватить трафик и получить конфиденциальные данные пользователя.

В реальной сети такой трафик могут читать:
- злоумышленники в одной Wi-Fi сети;
- администраторы сети;
- вредоносное ПО;
- атакующие через MITM-атаки.

---

## 3. Follow TCP Stream

Функция Follow TCP Stream позволяет увидеть весь диалог между клиентом и сервером в одном окне.

В ходе анализа были обнаружены:
- HTTP-запрос;
- логин пользователя;
- пароль;
- заголовки HTTP;
- содержимое POST-запроса.

Злоумышленник, перехвативший такой трафик, сможет полностью прочитать передаваемые данные и получить доступ к аккаунту пользователя.

## 4. HTTP vs HTTPS — сравнение

| Что наблюдаем | HTTP | HTTPS |
|---|---|---|
| IP-адреса сторон | Видны | Видны |
| Порт назначения | Видно 8080 | Видно 443 |
| Заголовки HTTP | Видны | Не видны |
| Содержимое запроса | Видно открытым текстом | Зашифровано |
| Логин и пароль | Видны | Не видны |
| Имя домена | Видно в Host | Частично видно через DNS/SNI |

## 5. DNS и приватность

Во время анализа DNS-трафика были обнаружены запросы к следующим доменам:
- github.com
- example.com
- google.com

Даже при использовании HTTPS DNS-запросы могут раскрывать информацию о посещаемых сайтах. Это означает, что провайдер или администратор сети всё равно может видеть, к каким доменам подключается пользователь.

Для повышения приватности используются:
- DNS over HTTPS (DoH);
- DNS over TLS (DoT);
- VPN.

## 6. Base64

Декодированная строка:

`dmFzeWE6TXlQYXNzd29yZDIwMjQ=` → `vasya:MyPassword2024`

Base64 не является шифрованием. Это лишь способ кодирования данных в текстовый формат.

Любой человек может легко декодировать такую строку с помощью команды base64 -d.

Поэтому HTTP Basic Authentication без HTTPS не обеспечивает защиту данных.

## 10. Контрольные вопросы

## 1. Что такое promiscuous mode и зачем он нужен снифферу?

Promiscuous mode (режим неразборчивого приёма) — это режим работы сетевого адаптера, при котором он принимает не только пакеты, адресованные ему, но и весь трафик, проходящий через сетевой сегмент. Снифферы используют этот режим для перехвата и анализа сетевого трафика. Без него адаптер обычно обрабатывает только пакеты, предназначенные конкретному устройству.

---

## 2. Чем отличается capture filter от display filter в Wireshark?

Capture filter применяется во время захвата трафика и определяет, какие пакеты будут записаны в файл захвата. Ненужные пакеты сразу отбрасываются.

Display filter применяется уже после захвата и используется для отображения только интересующих пакетов среди уже сохранённого трафика. При этом сами пакеты из файла не удаляются.

---

## 3. Что такое Follow TCP Stream и для чего используется?

Follow TCP Stream — функция Wireshark, которая собирает все пакеты одного TCP-соединения и отображает их в виде единого диалога между клиентом и сервером.

Она используется для анализа содержимого соединения, поиска передаваемых данных, логинов, паролей, команд и других сведений, переданных внутри TCP-сессии.

---

## 4. Почему Base64 в HTTP Basic Auth не является защитой? Что нужно использовать вместо?

Base64 не является шифрованием. Это лишь способ кодирования данных в текстовый формат.

Любой пользователь может мгновенно декодировать строку Base64 без знания какого-либо ключа. Поэтому логин и пароль в HTTP Basic Authentication легко восстанавливаются при перехвате трафика.

Вместо этого необходимо использовать HTTPS с протоколом TLS, который обеспечивает полноценное шифрование передаваемых данных.

---

## 5. Какие данные утекают даже при использовании HTTPS? Как это исправить?

Даже при использовании HTTPS остаются видимыми:

- IP-адреса источника и назначения;
- номер порта;
- DNS-запросы;
- информация о времени и объёме трафика;
- в некоторых случаях имя домена через SNI.

Для повышения приватности используются:

- DNS over HTTPS (DoH);
- DNS over TLS (DoT);
- VPN;
- Encrypted Client Hello (ECH).

---

## 6. Сисадмин говорит: «у нас свитч, а не хаб — значит сниффинг невозможен». Он прав?

Нет, это неверно.

Коммутатор действительно передаёт кадры только нужному получателю, поэтому обычный пассивный перехват усложняется. Однако существуют методы, позволяющие перехватывать трафик даже в сетях со свитчами:

- ARP-spoofing;
- MAC flooding;
- настройка зеркалирования портов (SPAN);
- компрометация самого коммутатора.

Поэтому наличие свитча само по себе не исключает возможность сниффинга.

## 7. Напишите фильтр tcpdump который перехватывает только DNS-запросы (UDP порт 53) от конкретного IP 192.168.1.50.

Команда:

sudo tcpdump -i eth0 'udp port 53 and host 192.168.1.50'

Данный фильтр будет перехватывать только DNS-трафик по UDP на порту 53, связанный с узлом 192.168.1.50.
